# Анализ Объединения Подмножеств Шрифтов


## Сравнение двух подходов

### Подход из EXAMPLE (iText PDF Optimizer)

#### Ключевые этапы объединения:

1. **Группировка шрифтов** (`FontMergingOptimizer.OptimizePdf`)
   - Использует `DocumentStructureUtils.GetSimilarDictionariesList()` для поиска похожих шрифтов
   - Применяет правила удаления subset-префиксов и уникальных полей для корректного сравнения
   - Создает группы эквивалентных шрифтов

2. **Сбор используемых глифов** (`UsedGlyphsFinder.FindUsedGlyphsInFonts`)
   - Использует `UsedGlyphsListener` для перехвата событий рендеринга текста
   - Создает карту `PdfFont -> UsedGlyphsFinder.FontGlyphs`
   - Собирает **только реально используемые глифы**, а не все глифы из шрифта

3. **Физическое объединение файлов шрифтов** (`TrueTypeFontUtil.MergeDocTrueTypeFonts`)
   - Для каждого шрифта извлекает GID (Glyph ID) всех используемых глифов
   - Вызывает `TrueTypeFont.Merge(dictionary2, fontName)` - внешний метод iText
   - **Это создает НОВЫЙ ФИЗИЧЕСКИЙ файл шрифта** с объединенным набором глифов
   - Возвращает `PdfStream` с новым файлом шрифта

4. **Объединение метрик ширины** (`TrueTypeMerger.MergeAndPutWidths`)
   - Создает объединение всех кодов глифов
   - Для каждого кода проверяет согласованность ширин
   - **При конфликте метрик - отказ от объединения**
   - Создает плотный массив `Widths` для диапазона [minCode, maxCode]

5. **Объединение ToUnicode** (`TrueTypeFontUtil.MergeAndPutToUnicode`)
   - Проверяет наличие/отсутствие ToUnicode во всех шрифтах (должно быть единообразно)
   - Для каждого шрифта парсит CMapToUnicode
   - Проверяет согласованность: одинаковые коды должны иметь одинаковые Unicode
   - **При конфликте Unicode - отказ от объединения**
   - Создает новый ToUnicode stream с объединенными маппингами

6. **Обновление PDF-словаря**
   - Клонирует базовый словарь шрифта
   - Удаляет subset-префикс и уникальные поля
   - Генерирует **новый** subset-префикс
   - Заменяет `FontFile2` на новый объединенный файл
   - Обновляет `Widths`, `ToUnicode`

### Подход в PdfFontOptimizer (текущий экспериментальный)

#### Что сделано правильно:

1. ✅ **Индексация шрифтов** (`IndexFonts`) - работает корректно
2. ✅ **Создание таблицы переназначения** (`CreateFontMappingTable`) - правильная группировка
3. ✅ **CID Remapping** (`CidRemapper.RemapCidsWithConflictResolution`) - есть попытка решить конфликты

#### Критические отличия и проблемы:

1. ❌ **НЕТ физического объединения файлов шрифтов**
   - Метод `CreateMergedFontDictionary` просто **копирует** `FontFile2` из первого шрифта
   - Не создается новый объединенный файл шрифта
   - **ПРОБЛЕМА**: Если глифа нет в первом шрифте, но есть во втором - он недоступен!

2. ❌ **Неправильная работа с метриками**
   - Просто копируется массив `W` из первого найденного CIDFont
   - Не происходит объединение и проверка согласованности
   - **ПРОБЛЕМА**: Неправильные ширины для глифов из других шрифтов

3. ⚠️ **ToUnicode обрабатывается, но неполно**
   - CID Remapper пытается создать новую карту ToUnicode
   - Но без физического объединения шрифтов это бесполезно
   - **ПРОБЛЕМА**: Новые CID ссылаются на несуществующие глифы

4. ❌ **Отсутствие проверок конфликтов**
   - Нет проверки совместимости метрик
   - Нет проверки согласованности ToUnicode
   - Нет отката при обнаружении несовместимости

## Почему текущий подход не работает

### Основная причина: отсутствие физического слияния

В iText Optimizer ключевым является вызов:
```csharp
byte[] fontStreamBytes = TrueTypeFont.Merge(dictionary2, fontName);
```

Этот метод **физически объединяет таблицы TrueType шрифтов** (glyf, loca, cmap, hmtx и т.д.), создавая новый файл шрифта, содержащий **все** необходимые глифы.

В текущем коде просто **копируется** FontFile2:
```csharp
var fontFile2 = originalFontDescriptor.Get(PdfName.FontFile2);
if (fontFile2 != null)
{
    newFontDescriptor.Put(PdfName.FontFile2, fontFile2); // ❌ Просто ссылка!
}
```

### Последствия:

1. **Отсутствующие глифы**: Если символ 'A' был в subset-2, но его нет в subset-1, то после "объединения" его не будет в результате
2. **Неправильные GID**: CID remapping создает новые коды, но в физическом файле шрифта нет соответствующих глифов
3. **Искаженный текст**: PDF-ридер не может найти глифы по новым CID и показывает случайные символы или пропуски

## Что нужно исправить

### Критически важно:

1. **Реализовать физическое объединение файлов шрифтов**
   - Либо использовать `TrueTypeFont.Merge()` из iText (если доступен)
   - Либо реализовать собственную логику слияния TrueType таблиц
   - **Без этого объединение невозможно**

2. **Объединить и проверить метрики**
   - Собрать массив `W` (widths) из всех источников
   - Проверить согласованность: одинаковые CID должны иметь одинаковые ширины
   - При конфликте - отказаться от объединения этой группы

3. **Правильно объединить ToUnicode**
   - Проверить наличие ToUnicode во всех источниках (должно быть единообразно)
   - Извлечь и объединить маппинги
   - Проверить на конфликты: одинаковые CID -> одинаковые Unicode
   - При конфликте - отказаться от объединения

### Желательно:

4. **Добавить валидацию**
   - Проверка типа шрифта (CFF не поддерживается)
   - Проверка совместимости кодировок
   - Логирование причин отказа от объединения

5. **Оптимизировать CID remapping**
   - Сейчас он работает, но не нужен при правильном физическом объединении
   - В iText подходе физическое объединение сохраняет оригинальные GID
   - CID remapping нужен только если физическое объединение невозможно

## Вывод

**Текущий экспериментальный код пытается решить проблему на уровне PDF-структуры**, не затрагивая физические файлы шрифтов. Это фундаментально неверный подход.

**iText Optimizer решает проблему физическим объединением файлов шрифтов**, а затем обновляет PDF-структуру для ссылки на новый объединенный шрифт.

Для корректной работы нужно:
1. **Либо использовать iText API** для физического объединения (если доступен)
2. **Либо реализовать собственный merger** для TrueType таблиц (сложно, требует глубоких знаний формата)
3. **Либо отказаться от физического объединения** и просто удалить дубликаты (без объединения глифов из разных subset'ов)

Третий вариант не решает исходную задачу (объединение неполных subset'ов), но может уменьшить размер файла за счет устранения дублирования одинаковых subset'ов.
